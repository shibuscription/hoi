# あっち向いてHOI

## 概要
「あっち向いてHOI」は、現在地は把握できていても、自分がいまどちらを向いているのか分からず、最初の一歩を決めにくい人向けの補助アプリです。地図そのものを読む力を前提にするのではなく、実際に見えている景色と周辺のストリートビューを見比べることで、手動で向きを合わせられることを重視します。

本アプリは、フル機能の地図アプリや経路案内アプリを置き換えるものではありません。役割としては「現在地は分かるが向きが分からない」という初動の迷いを解消することに絞り、向きが確定した後の詳細な地図閲覧やナビゲーションは Google マップなど既存アプリへ引き継ぐ想定です。

## 解決したい課題
地図アプリで現在地のピンが表示されても、現実の風景と地図上の向きが頭の中で結びつかず、どちらに歩き始めればよいか判断できないことがあります。本アプリは、特にその「最初の一歩が決められない」状態の支援を目的とします。

主に想定している課題は以下です。

- 現在地は分かるが、自分がどちらを向いているか分からない
- 地図と実景を対応付けるのが苦手で、頭の中で回転させて理解しづらい
- 駅前や交差点などで、進み始める方向を決めるまでに時間がかかる

## 基本アイデア
基本画面は、上側にユーザーが撮影した実景画像、下側に現在地近傍のストリートビューを表示する構成を想定します。ユーザーは両者を見比べながら、いま自分が向いている方向に近い景色になるよう手動で合わせます。

向きが一致したと判断できた時点で、アプリ内で向きを確定し、その結果を地図表示へ反映します。ここで重要なのは、方位センサーや画像認識による自動推定ではなく、ユーザー自身が「いま見えている景色」と「ストリートビュー上の景色」を照合して決める点です。

## 想定ユースケース
本アプリは、地図を読むこと自体が得意な人よりも、地図と実景の対応付けに負荷を感じやすい人の初動支援を主な対象とします。特に、目的地への経路が分からないというより、「まずどちらへ向けばよいか」が分からない場面で価値を出す想定です。

代表的なユースケースは以下です。

- 駅前で降りた直後に方向感覚を失い、どちらに歩き始めればよいか分からない
- 観光地や初めて訪れた場所で、周囲の景色は見えているが地図の向きがつかめない
- 地図アプリを開いても、自分の向きが曖昧なせいで最初の移動方向を決められない

## 想定画面フロー
Phase 1 では、利用者が迷わず一連の操作を進められるよう、以下のような直線的なフローを想定します。

1. 位置情報取得
2. カメラ起動・撮影
3. ストリートビュー照合
4. 向き確定
5. 地図表示
6. Google マップへ引き継ぎ

それぞれの画面は、複雑な分岐を避けつつ、「今やること」が明確に分かる UI を目指します。特に照合画面では、撮影画像とストリートビューを見比べる操作に集中できる構成が重要になります。

## Phase 1 の仕様メモ
Phase 1 は MVP として、向き合わせの成立までを最短で体験できることを重視します。自動認識や高機能なナビゲーションは入れず、最低限の入力・照合・確定・引き継ぎまでをひとつながりで実現する方針です。

現時点の MVP 範囲は以下を想定します。

- 位置情報取得
  - スマホの位置情報を利用して現在地座標を取得する
- カメラ撮影
  - ユーザーが現在見えている風景や目印を撮影する
- ストリートビュー表示
  - 現在地近傍のストリートビューを取得して表示する
- 手動照合
  - ユーザーが撮影画像とストリートビューを見比べて向きを合わせる
- 向き確定
  - ユーザー操作で向きを確定し、アプリ内状態として保持する
- 地図表示
  - 現在地と確定した向きを地図上に表示する
- Google マップへの遷移
  - 必要に応じて Google マップへ引き継げる導線を提供する

なお、方位センサーや傾きセンサーは Phase 1 では使わない前提です。また、画像認識による自動一致判定も行わず、照合の主体はあくまでユーザーに置きます。

## Phase 1 の完了条件
Phase 1 は「最小構成でも、向きが分からない状態から Google マップへ引き継げる状態まで一通り通せること」を完了の目安とします。単に画面が存在するだけではなく、最低限の成功パスと主要エラー時の案内が成立していることを条件に含めます。

現時点では、以下を満たせたら Phase 1 完了とみなす想定です。

- 現在地を取得できる
  - ユーザー操作により現在地座標を取得し、後続処理で利用できる
- 写真を 1 枚撮影して固定表示できる
  - カメラを起動し、その場の実景画像を撮影して照合画面に保持できる
- 近傍 Street View を表示できる
  - 現在地近傍の Street View を取得し、照合対象として表示できる
- ユーザーが手動で向きを合わせて確定できる
  - 自動認識に頼らず、ユーザー操作で向きを合わせて確定できる
- 確定後に地図上へ現在地と向きを表示できる
  - 現在地と確定した向きが地図上で確認できる
- Google マップへ 2 系統で遷移できる
  - 「地図で開く」と「この向きの景色で開く」の 2 導線を提供できる
- 最低限のエラー表示ができる
  - Street View 未取得時、位置情報権限拒否時、カメラ権限拒否時などに処理が止まった理由を表示できる

この完了条件は、体験の成立確認を主目的とした暫定定義です。UI の磨き込みや閾値調整などは、Phase 1 完了後の改善対象として切り分ける想定です。

## 非機能要件
Phase 1 の段階では、対象環境と利用前提を絞ることで、実装と検証のブレを減らす方針です。ここでの非機能要件は、将来の拡張を縛るためではなく、まずどこで安定して動くべきかを明確にするための整理として置きます。

現時点での想定は以下です。

- スマホ縦画面を主対象とする
  - 片手操作と実景確認を前提に、まずは縦持ちで使いやすい画面構成を優先する
- モバイルブラウザでの利用を主対象とする
  - 当面はスマホのブラウザ利用を前提とし、PC 向け最適化は優先しない
- HTTPS 環境を前提とする
  - 位置情報やカメラのブラウザ機能を安定して使うため、配信環境は HTTPS を前提とする
- 位置情報とカメラの権限が必要
  - どちらかが許可されない場合、主要機能は成立しない前提で設計する
- 通信状態により Street View や地図表示に失敗する場合がある
  - オンライン前提の機能が多いため、通信不安定時の失敗表示を考慮する
- 当面はネイティブアプリ化を前提にしない
  - Phase 1 は Web アプリとして成立するかを優先し、OS 固有実装は持ち込まない
- PWA 化は今後の検討事項とする
  - ホーム画面追加やキャッシュ戦略は有力候補だが、初期段階では必須条件にしない

## 利用する外部 API / 権限整理
Phase 1 では、ブラウザ標準機能と Google Maps 系の API を中心に構成する想定です。ここでの記載は技術選定の最終確定ではなく、実装に着手する際の依存関係と責務の見取り図として整理しています。

- Geolocation API
  - 現在地取得のために使用する想定です。ユーザーの明示的な権限許可が必要であり、精度や取得速度は端末・通信環境に左右されます。
- `MediaDevices.getUserMedia`
  - カメラ起動と撮影プレビューのために使用する想定です。HTTPS 環境が前提であり、端末やブラウザによって利用可能なカメラや挙動に差が出る可能性があります。
- Google Maps JavaScript API
  - 地図表示や向き確定後の現在地表示に使用する想定です。API キー管理や利用制限、料金体系は事前に整理が必要です。
- Street View Service
  - 現在地近傍のパノラマ取得に使用する想定です。必ずしも現在地そのもののパノラマが返るとは限らず、取得地点との差分を UI 上でどう扱うかが重要になります。
- Google Maps URL 連携
  - 「地図で開く」「この向きの景色で開く」の外部遷移に使用する想定です。遷移先で再現できる情報量には限界があるため、URL で渡せる値と体験上の期待値を合わせる必要があります。

権限面では、少なくとも位置情報とカメラの利用許可が必要です。方位センサーや傾きセンサーは Phase 1 では使わない前提のため、関連権限やセンサー API は現時点では対象外とします。

## ストリートビュー利用時の考慮点
ストリートビューは本アプリの中核要素ですが、取得できる場所とできない場所があり、また取得できたとしても実際の現在地と完全に一致するとは限りません。そのため、単純に「表示できたから使える」とは扱わず、利用可否や信頼度を明示する方針が必要です。

現時点での考慮方針は以下です。

- ストリートビューが取得できない場所では、無理に代替せずエラー表示で終了してよい
- ストリートビュー座標と現在地座標にずれが生じる可能性がある
- 現在地座標とストリートビュー座標の距離差をもとに、信頼度の参考値を出す
- 距離差が大きい場合や実景と一致しにくい場合は、注意表示を出す
- 一致しない場合にユーザーへ過剰な試行を求めず、失敗として終えられる設計にする

この信頼度は、機械的な正解判定ではなく、「この照合結果をどの程度信用してよいか」を補助的に示す情報として扱う想定です。

暫定ルールとしては、現在地座標と Street View 座標の差が小さい場合を「高」、少し離れているが参考にはなる場合を「中」、離れすぎていて現在地そのものの景色ではない可能性がある場合を「低」とする整理が考えられます。数値しきい値を置く場合も、初期段階では仮置きとし、実地確認を通じて今後調整する前提にします。

## 想定エラーケース一覧
Phase 1 では、すべての失敗を詳細にリカバリーするよりも、「なぜ進めないのか」を利用者に伝えられることを優先します。以下は現時点で代表的に想定しているエラーケースです。

- 位置情報が取得できない
  - 端末設定、通信状況、取得タイムアウト、ブラウザ実装差などにより、現在地の取得自体が完了しない場合があります。
- 位置情報権限が拒否される
  - ユーザーが位置情報利用を許可しないと、現在地基準で後続処理を進められません。
- カメラ権限が拒否される
  - カメラ利用が許可されない場合、実景撮影と照合体験が成立しないため、主要導線は継続できません。
- 撮影に失敗する
  - カメラ起動後でも、端末制約やブラウザ挙動、ユーザー操作中断などにより撮影画像を取得できない場合があります。
- Street View が取得できない
  - 近傍にパノラマが存在しない場所や API 応答失敗時には、照合対象が用意できません。
- Street View は取得できるが現在地から離れすぎていて信頼度が低い
  - 近傍検索で返った地点が実際の立ち位置とずれていると、照合結果の参考性が下がるため注意表示が必要です。
- 地図表示や Google マップ連携に必要な値が不足している
  - 現在地、向き、遷移用パラメータのいずれかが欠けると、地図表示や外部遷移を安全に成立させられません。

これらのケースでは、単に処理失敗とするのではなく、再試行できるのか、権限設定の見直しが必要なのか、その場では利用できないのかが分かる案内文を出す方針が望まれます。

## Google マップ連携の考え方
本アプリの役割は、向きが分からず動き出せない状態を解消するところまでとし、その後の地図閲覧や経路案内は既存の地図アプリへ委ねる方針です。これにより、本アプリは責務を絞りつつ、既存サービスの強みを活用できます。

現時点では、以下のような連携を想定します。

- 地図で開く
  - 現在地または目的地周辺を Google マップで開く
- この向きの景色で開く
  - 確定した向きに近い景色・視点へ遷移しやすい形で Google マップへ引き継ぐ

連携方式の詳細は今後の技術選定に依存しますが、「向き合わせまではあっち向いてHOI、その後の移動支援は Google マップ」という役割分担は維持する想定です。
